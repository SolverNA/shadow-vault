# Инструкции для Claude Code

## Работа через суб-агентов

**КРИТИЧЕСКИ ВАЖНО:**

- **ВСЕГДА** выполнять работу через суб-агентов. Координатор (основной Claude) **НИКОГДА не лезет в код сам**.
- Координатор только: изучает задачу, планирует, запускает суб-агентов, сводит их отчёты, общается с пользователем.
- Запрещено координатору самому читать/править исходный код, чтобы не забивать контекст кодом — для любого изучения и изменения кода запускается суб-агент.
- Суб-агенты возвращают **сжатые отчёты** (выводы, пути `file:line`), а не дампы кода.

## Работа с Git и версионированием

- **ОБЯЗАТЕЛЬНО** версионировать все изменения через Git
- Писать понятные commit messages на русском языке
- Работать в отдельных ветках для новых фич, мержить в main после завершения

### Кто и как коммитит

- **Суб-агенты НЕ коммитят сами.** Никогда не выполнять `git commit` / `git add` внутри суб-агента.
- Суб-агент завершает работу и **выдаёт готовые git-команды** (текстом), которые **пользователь вводит сам** вручную.
- Координатор передаёт эти команды пользователю и ждёт, пока пользователь закоммитит сам.

### Один коммит — одно действие

- Коммиты **строго по одному действию / одной задаче**.
- **ЗАПРЕЩЕНЫ** мульти-коммиты, которые «исправляют всё сразу» в одном коммите.
- Каждое логическое изменение = отдельный коммит со своим конкретным сообщением строго по задаче.
- Если сделано несколько правок — выдать пользователю **несколько отдельных команд коммита**, по одной на действие.

## Документация

- **НЕ создавать** новые файлы документации (*.md) без явного запроса
- **НЕ разводить** документацию по разным файлам
- **ЕДИНСТВЕННЫЙ** основной файл документации: `README.md`
- При изменении проекта **ДОПОЛНЯТЬ** README.md, а не создавать новые файлы
- При переработке архитектуры **ИЗМЕНЯТЬ** существующие разделы в README.md
- README.md содержит: ТЗ, идеи, архитектуру проекта

## Коммуникация действий

**ОБЯЗАТЕЛЬНО** описывать каждое свое действие перед выполнением:

- При изменении кода: "Изменяю [название системы/модуля] для [цель/причина]"
- При коммите: "Коммичу изменения: [описание]"
- При копировании файлов: "Копирую [файл] в [место] для [цель]"
- При изучении проекта: "Изучаю проект, читаю [конкретный файл]"
- При любой операции: четко называть что делаю и зачем

## Процесс работы с задачами

**КРИТИЧЕСКИ ВАЖНО:**

1. **Выслушать** задачу от пользователя
2. **Пересказать** своими словами как понял задачу, что планирую сделать
3. **Дождаться подтверждения** что все правильно понял
4. **ТОЛЬКО ПОСЛЕ ОДОБРЕНИЯ** приступать к действиям
5. Никогда не начинать работу без явного подтверждения понимания

## Пример правильного workflow

```
Пользователь: добавь логирование в connection manager
Claude: Понял, нужно добавить логирование в модуль connection manager (core/src/connection/mod.rs). 
        Буду логировать события подключения/отключения и ошибки. Правильно?
Пользователь: да
Claude: Изменяю connection manager для добавления логирования событий.
        [выполняет изменения]
        Коммичу изменения: "feat(connection): add event logging"
```

## Работа с командами, требующими пароль

**ОБЯЗАТЕЛЬНО** при невозможности выполнить команды из-за требования пароля:

1. **Вывести** пользователю команды, которые нужно выполнить
2. **Попросить** выполнить их и прислать вывод: "Выполни эти команды и скинь мне вывод, я продолжу"
3. **Продолжить** выполнять другие команды, не требующие пароля, пока жду ответа
4. **После получения** вывода от пользователя - продолжить работу с учетом результатов

Пример:
```
Claude: Не могу выполнить `sudo apt install package` - требуется пароль.
        Выполни эту команду и скинь мне вывод, я продолжу.
        Пока продолжаю работу с другими задачами...
```

## Что НЕ делать

- ❌ Не создавать CONTRIBUTING.md, ARCHITECTURE.md, CHANGELOG.md без запроса
- ❌ Не начинать работу без подтверждения понимания задачи
- ❌ Не делать изменения молча, всегда озвучивать действия
- ❌ Не разбрасывать документацию по файлам
- ❌ Координатору НЕ лезть в код самому — только через суб-агентов
- ❌ Суб-агентам НЕ коммитить самим — выдавать команды пользователю
- ❌ НЕ делать мульти-коммиты «всё сразу» — строго один коммит на одно действие
